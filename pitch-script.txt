"Most urban data is currently 'dead' — trapped in siloed spreadsheets and static
maps that fail to drive an instant economy. PulsePath is designed to bridge this
gap. As a Geography and Data Science student at the University of Washington, I've

What PulsePath does in three steps:

  INPUT  → Conversational criteria intake powered by Gemini 2.0 Flash NLU
  ENGINE → Weighted geospatial scoring + ArcGIS Maps SDK v5 rendering
  OUTPUT → Ranked, actionable neighborhood decisions

Tech stack:
  - React 18 + TypeScript (front-end)
  - ArcGIS Maps SDK v5 (spatial rendering)
  - Gemini 2.0 Flash (NLU / intent extraction)
  - criteriaScoring.ts (weighted spatial ranking engine)

HOW THE PIPELINE WORKS:

  ① NLU (Natural Language Understanding)
     Gemini 2.0 Flash extracts intent, location anchors, housing type, budget,
     and priority weights directly from free-form user input.

  ② SCORING
     criteriaScoring.ts applies weighted spatial analysis across four dimensions:
     transit access, safety index, wellness density, and affordability.

  ③ MAP RENDERING
     ArcGIS v5 renders ranked neighborhoods with color-coded score markers
     that update in real time as the user refines their preferences.

LIVE EXAMPLE:

  User:   "I want somewhere safe, walkable, near coffee shops, under $1,800/mo"

  Gemini: { safety: 0.35, transit: 0.28, wellness: 0.22, affordability: 0.15 }

  Result: Capitol Hill #1 · Fremont #2 · Queen Anne #3

The model also implements a 3-tier fallback for reliability:
  Gemini 2.0 Flash → Gemini 2.0 Flash-Lite → Gemini 1.5 Flash → Regex fallback

Key improvements made during this development cycle:

  1. FROM STATIC TO LIVE
     Refactored beyond mockData.ts. ArcGIS Feature Services now provide 2-way
     sync between user interaction and live map state. Markers update without
     a page reload. Data flows both directions — user actions update the map,
     and map events update the UI.

  2. ENHANCED CRITERIA ENGINE
     criteriaScoring.ts now processes complex spatial weights, including:
       - Proximity to transit-oriented development (TOD)
       - Neighborhood resilience scoring
       - Real commute buffer geometry
       - Wellness venue density (parks, gyms, clinics)
     This produces significantly more accurate and personalized rankings.

  3. REFINED STATE MANAGEMENT
     The Intake Panel (where users express their preferences) and the geospatial
     scoring engine are now tightly coupled via React Context. Every preference
     change triggers immediate map recalibration. The isSending state flag
     prevents race conditions during async Gemini NLU calls, ensuring results
     are always consistent with the most recent input.

  4. CITY-AGNOSTIC ARCHITECTURE
     Modular data ingestion proved via config.setMarket(). Swap out ArcGIS
     Feature Layer URLs and the entire scoring, ranking, and map system adapts
     to a new city with zero core logic rewrites.

     Example: config.setMarket('NYC_DOT_FEED'); intelligence.recalibrate();

     This makes the Seattle Hub a template — not a one-off. NYC, London, Tokyo,
     or any market with open geospatial feeds can be onboarded in minutes.

Q&A CHEAT SHEET — TECHNICAL SPECIFICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: How does the NLU actually work?
A: Users type a free-form message in the Intake Panel. The message goes to Gemini
   2.0 Flash via the REST API. A structured prompt asks Gemini to return JSON
   with extracted fields: intent, housing type, bedrooms, max budget, location
   anchor, and priority weights (safety, transit, wellness, affordability). That
   JSON is merged into the app's SearchCriteria state object, which triggers a
   re-score and map re-render. If Gemini is unavailable, a regex fallback
   pattern-matches common phrases.

Q: How does the scoring engine work?
A: criteriaScoring.ts takes the SearchCriteria object and iterates over all
   neighborhoods. Each neighborhood has scores for safety, transit, wellness,
   and affordability. The function applies the user's priority weights as a
   weighted average, producing a composite score 0–100. Neighborhoods are ranked
   by this score on both the list view and the ArcGIS map markers.

Q: How does the map connect to the data?
A: PulseMap.tsx uses @arcgis/core v5. It creates a GraphicsLayer and places
   one Graphic per neighborhood with a SimpleMarkerSymbol (green ≥70, yellow
   ≥45, red <45) and a TextSymbol label. A hitTest handler on click selects
   the neighborhood and updates the Context state. The GraphicsLayer refreshes
   whenever the scored neighborhoods prop changes.

Q: Can this actually scale to other cities?
A: Yes. The scoring engine, NLU prompts, and React components are all decoupled
   from Seattle-specific data. The only city-specific pieces are the ArcGIS
   Feature Layer URLs and the neighborhood seed data. Swapping those via a
   config object (config.setMarket) is the only step required for a new city.

Q: What's the Zillow integration?
A: The dashboard's residential layer surfaces top-ranked neighborhoods with an
   outbound link constructed from the neighborhood name + city. Clicking it
   opens a pre-filtered Zillow search for that area. Future versions would use
   the Zillow Bridge API for direct listing data ingestion.

Q: Is the Gemini API key secure?
A: The key is read from the VITE_GEMINI_API_KEY environment variable at build
   time via import.meta.env. It is never hardcoded. In production, this would
   be proxied through a server-side endpoint so the key is never exposed to
   the browser.
